Workflow

The goal of this document it to outline the workflow of bugs and features at Social Tables from the point of conception or discovery to deployment with a detailed focus on Project Workflow and Engineering Workflow parts of the process.

Overview

Feature requests come from any where and everywhere, but no matter where the come requests that are going to make it into the app find their way to the top of the funnel to be outlined below.

Bug requests on the other hand skip the Product Workflow steps that are briefly outlined below and are inserted into our work flow at the start of Project Workflow.

  1. Product Workflow
    1. Customer Development: This is when a feature is expressed to a customer with the hopes of better understanding what the customer really wants.
    2. Design: Here features are turned into visuals that can be shown to customer for more development or to engineers as guides.
  2. Project Management
    1. Storyification: This is where a feature or is broken into user stories by the Product Team
    2. Pointing: This is where engineering assigns difficulty point to each story.
    3. Prioritization: This is where product orders stories based on a combination of business value and engineering difficulty.
    4. Sprint Planning: This is where engineering and product agree on what the goal is to deliver in the next sprint.
  3. Engineering Workflow
    1. Queued: This is where stories sit while waiting to be worked on and part of the current sprint.
    2. Active: This is where stories live while an engineer is working on them.
    3. Code Review: This is where the code of a story is reviewed by a member of the team that did not contribute to the new code.
    4. Functionality Review: This is where the functionality of the feature is reviewed either by Product or Engineering.
    5. Ready: This is where a feature sits while waiting to be deployed
    6. Deployed: This is where a feature sits once it is deployed.
